# Бизнес-процессы приложения

Этот файл описывает текущее поведение приложения на уровне бизнес-логики.

## 1. Пользователи и доступ

- Пользователь работает только после регистрации и входа по email/паролю.
- Доступ к доске есть только у тех пользователей, которые добавлены в нее как участники.
- Участник доски всегда имеет одну из ролей:
  - `admin`
  - `editor`
  - `viewer`

## 2. Доски

- Пользователь может создавать доски.
- Создатель доски автоматически становится ее участником с ролью `admin`.
- На дашборде пользователь видит только те доски, в которых он состоит.
- `admin` может:
  - переименовывать доску
  - удалять доску
  - приглашать участников
  - менять роли участников
  - удалять участников из доски
- Пользователь не может удалить самого себя из доски через управление участниками.

## 3. Участники доски

- Приглашение в доску происходит по email.
- Если пользователь найден, он добавляется в доску.
- По умолчанию приглашенный пользователь получает роль `editor`.
- Роли используются так:
  - `admin` управляет доской и может выполнять любые действия с колонками и задачами
  - `editor` может работать с колонками и задачами
  - `viewer` не может создавать и редактировать задачи или колонки, но может работать со своими задачами по правилам ниже

## 4. Колонки

- Колонка принадлежит конкретной доске.
- У колонки есть:
  - название
  - тип
  - позиция
- Поддерживаются типы колонок:
  - `backlog`
  - `in_progress`
  - `done`
- `admin` и `editor` могут:
  - создавать колонки
  - переименовывать колонки
  - менять тип колонки
  - менять порядок колонок
  - удалять колонки
- Если удаляется колонка, ее задачи переносятся в первую оставшуюся колонку.
- Если удаляется последняя колонка доски, приложение автоматически создает новую колонку:
  - название: `Бэклог`
  - тип: `backlog`

## 5. Задачи

- Задача принадлежит колонке.
- У задачи есть:
  - название
  - описание
  - создатель
  - исполнитель
  - позиция внутри колонки
- Создавать задачи могут только `admin` и `editor`.
- Редактировать задачу могут только `admin` и `editor`.
- При создании или редактировании задачи можно указать исполнителя или оставить задачу без исполнителя.
- Удалять задачу могут пользователи, у которых есть право редактирования доски.

## 6. Перенос задач

- `admin` и `editor` могут переносить любые задачи между колонками и менять их порядок.
- Исполнитель задачи может переносить только свою задачу.
- Если задача не назначена и находится в колонке типа `backlog`, любой участник доски может забрать ее в работу:
  - для этого он переносит задачу из `backlog` в колонку другого типа
  - в момент такого переноса задача автоматически назначается на этого пользователя
- После назначения задачи пользователь может переносить только ту задачу, где он указан исполнителем.
- Пользователь не может переносить чужую назначенную задачу, если он не `admin` и не `editor`.

## 7. Комментарии к задачам

- Комментарии доступны внутри задачи.
- Комментарий может создать любой участник доски, который видит эту доску.
- Редактировать комментарий может только его автор.
- Удалять комментарий может только его автор.
- В комментариях используется обычный текст без rich text-форматирования на уровне бизнес-логики.

## 8. Дашборд и статистика

- На дашборде пользователь видит:
  - список своих досок
  - количество досок
  - общее количество задач в доступных досках
  - общее количество уникальных участников в доступных досках

## 9. Демо-данные

- В проекте есть сидер для ручного тестирования.
- Он создает:
  - пользователей с разными ролями
  - несколько досок
  - колонки разных типов
  - задачи
  - комментарии
- Сидер повторяемый: повторный запуск не должен плодить дубликаты основных сущностей.

## 10. Что отсутствует в текущей версии

- Нет realtime-обновлений и broadcast-событий.
- Нет входа через внешние OAuth-провайдеры.
- Нет email verification, two-factor authentication и confirm password flow.
- Нет отдельного service layer: бизнес-логика находится в контроллерах и request-классах.
